SDD Harness

软件开发全流程框架

从需求澄清、PRD 设计、原型制作到开发计划的全流程规范化框架。 三端统一、路径守门、阶段门禁，让软件开发真正可控、可追溯、可交付。

项目概述

为什么软件开发需要更好的框架

🔴 现状问题

传统软件开发流程混乱：需求不清晰就开干、设计文档缺失、 代码和文档脱节、项目状态不可追溯。多端工具（Codex、Cursor、Claude Code） 各自为政，规则不统一，经验无法沉淀。

🟢 解决方案

SDD Harness 是一套全流程软件开发框架。从需求澄清到开发计划， 每一步都有明确规范。路径守门防止误写入，阶段门禁确保节奏可控， Git 标签支持随时回滚。三端统一协议，一处修改处处生效。

核心特性

SDD Harness 的与众不同之处

🔄

四路顶层路由

新建项目、已有项目修整、导入已有项目、删除项目—— 四个入口覆盖全生命周期，意图清晰分流。

🚦

路径守门机制

所有业务文件必须写入 `Projects_Repo//`， Harness 根目录严禁业务产物，零误写入。

✅

阶段门禁确认

每个阶段完成后必须用户确认才能进入下一阶段。 禁止跳步，禁止静默推进，用户始终掌控节奏。

⏮️

Git 版本回滚

每个阶段和每次关键操作自动打标签。 随时回滚到任意历史状态，大胆实验。

🎯

三端统一入口

Codex、Claude Code、Cursor 三端统一协议。 核心流程只维护在 harness-core/，平台目录只做适配。

📦

项目注册表管理

统一的项目注册表，自动管理 active_project_id。 脚本初始化、状态同步、一键切换项目。

产品设计工作流

从需求澄清到开发计划，五阶段完整闭环

🎯 R 需求澄清
↓
📝 A PRD 初稿
↓
🎨 B1 设计规范
↓
🖼️ B2 原型验证
↓
📋 C 开发计划
阶段说明：
  • R：需求澄清 + 竞品调研 + 视觉证据沉淀
  • A：PRD 初稿（A1 功能列表 → A2 用户故事 → A3 数据契约 → A4 业务规则 → A5 非功能需求）
  • B1：Design Context + Design Spec（UI 设计规范）
  • B2：原型架构选择 + HTML/Pencil/Stitch 原型 + 验证
  • C：PRD 定稿 + api-contracts.md + Plan.md

设计理念

SDD Harness 背后的设计原则

核心规则单一维护

SDD 核心流程只维护在 harness-core/，平台目录（.codex/.cursor/.claude）只做适配。 一处修改，三端同步，避免规则分裂。

禁止跳步，门禁优先

每个阶段必须用户确认才能进入下一阶段。 "继续"只在明确询问后生效，防止误判意图导致跳步。

路径守门，零误写入

所有业务文件必须写入 Projects_Repo//， Harness 根目录严禁业务产物。路径守门，零误写入。

精简产出，禁止文件膨胀

非必要不新增文件。过程中的调研、草稿、中间结论在对话中完成， 最终结论合并写入核心文件。
