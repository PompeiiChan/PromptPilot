# Prompt PRD 结构规范

本文件定义 Prompt PRD（`docs/prd.md`）的标准结构。PRD 是 Prompt Harness 全链路的**单一事实来源**：

```
Requirements Clarifier  → 按本规范产出 prd.md
Prompt Writer           → 按本规范从 prd.md 取要求写 Prompt
Eval Generator          → 按本规范从 prd.md 取要求造评测集
Eval Designer           → 按本规范从 prd.md 取要求设评测框架
```

只要 PRD 的字段契约稳定，下游三个 Agent 就能各取所需、互不冲突。

## 定位：面向 Assistant / System Prompt

本 Harness 主要生产 **Assistant / System Prompt**——给一个 AI 助手/角色配置长期行为规范（类似 GPTs 的 system prompt）。因此 PRD 的重心是：

- **角色**：这个助手是谁
- **能力边界**：能做什么、不能做什么
- **行为准则**：必须遵守的规则和红线
- **多轮行为**：在连续对话里如何保持一致

而不是单次任务的输入输出映射。

---

## 一、核心概念：主需 vs 次需

**这是 PRD 必须供给下游的最重要区分。** 整个评测体系（`overall-scoring` 的"主需满足度"、`rubric-scoring` 的考点权重）都建立在它之上。PRD 不明确拆分主需/次需，下游评测就无法对齐。

| | 定义 | 不满足的后果 |
|---|------|------------|
| **主需** | 用户使用这个 Assistant 时，必须被完成的核心动作或必须获得的核心信息 | 直接导致回复不可用 |
| **次需** | 核心之外的约束条件、额外要求，如字数、风格、格式、特定偏好 | 价值有损，但不一定不可用 |

PRD 必须在 Section 5 显式列出主需清单和次需清单。

---

## 二、PRD 标准结构

PRD 用 Markdown 编写，章节顺序固定。标 **[必填]** 的章节缺失则不得通过 Phase 1 门禁；标 **[条件]** 的按交互形态决定；标 **[选填]** 的可留空但要保留标题。

```markdown
# Prompt PRD：<prompt_name>

## 0. 元信息 [必填]
- prompt_id：<kebab-case>
- 交互形态：单轮 | 多轮 | 两者兼容    ← 驱动 Section 8 是否必填
- PRD 版本：v1
- 最后更新：YYYY-MM-DD

## 1. 目标与背景 [必填]
这个 Assistant 要解决什么问题、服务什么场景。一段话讲清"为什么需要它"。

## 2. 角色定义 [必填]
- 身份：这个助手是谁（如"资深法律咨询顾问"）
- 专业领域：覆盖的知识范围
- 视角立场：以什么角度作答（如"始终站在用户利益一方"）

## 3. 使用场景与用户画像 [必填]
- 使用者：谁会和这个助手对话
- 典型场景：在什么情况下使用
- 典型 query 示例：用户实际会问什么（给 2-3 个真实例子）

## 4. 能力边界 [必填]
- 能做：明确列出助手应当胜任的任务
- 不能做：明确列出超出范围的任务
- 超界处理：遇到能力外的请求如何回应（拒答话术、转介建议等）

## 5. 需求拆分 [必填]
### 主需（核心，缺失即不可用）
1. <主需 1>
2. <主需 2>
### 次需（约束，影响价值但不致命）
1. <次需 1，如"每次回答控制在 300 字内">
2. <次需 2，如"语气专业不口语化">

## 6. 行为准则与红线 [必填]
- 行为约束：必须遵守的规则（逐条，可验证）
- 安全红线：绝对不能触碰的内容（涉政、违法、医疗/金融合规等）
- 真实性要求：不确定信息如何处理（如"不确定就声明，禁止编造")

## 7. 输出规格 [必填]
- 输出格式：纯文本 / Markdown / 结构化（给出模板）
- 长度约束：具体区间（不用"简洁"等模糊词）
- 结构要求：是否需要分点、是否需要总结等

## 8. 多轮行为规范 [条件：交互形态含多轮时必填]
- 上下文记忆：需要记住前轮的什么信息
- 人设一致性：多轮中如何保持角色和语气稳定
- 个性化：如何根据用户前轮画像/偏好调整后续回答
- 话题推进：是否需要主动推进对话、何时追问
- 轮间一致性：如何避免与前轮回答矛盾

## 9. 语气与风格 [必填]
- 语气：专业 / 友好 / 严肃 / 简洁……
- 风格约束：是否用 emoji、是否口语化、称呼方式等

## 10. 备选模型 [必填]
- 主力模型：<model id>
- 对比模型：<model id>（如有）
- 选型理由：为什么选这些模型（可简述）

## 11. 成功标准 [必填]
用自然语言描述"什么样的表现算好"：
- 达到几分算可接受（对接 overall-scoring 五档：通常 ≥3 可用，≥4 好用）
- 最关心哪些质量维度（深度？真实性？特异性？）
- 哪类错误绝对不能出现

## 12. 典型示例 [必填]
### 理想样例
**用户输入**：<query>
**理想回复**：<response 或其关键特征>
**为什么好**：<说明>
### 反例（不好的回复）
**用户输入**：<query>
**糟糕回复**：<response>
**问题所在**：<说明>
```

## 三、字段 → 下游消费映射表

这张表是本规范的核心：每个字段都标明被谁消费、怎么用。写 PRD 的人据此知道每个字段为何重要；下游 Agent 据此知道去 PRD 哪里取数据。

| PRD 章节 | Prompt Writer 用法 | Eval Generator 用法 | Eval Designer 用法 |
|---------|-------------------|---------------------|-------------------|
| 2 角色定义 | 写 Prompt 的角色段 | — | — |
| 3 场景/画像 | 校准语气和深度 | 造贴近真实场景的用例 | — |
| 4 能力边界 | 写边界处理指令 | 造"超出能力"的负面用例 | 设边界考点 + 超界扣分点 |
| 5 主需 | 写核心指令 | 造覆盖每条主需的用例 | 设主需考点（高权重 9-10） |
| 5 次需 | 写约束 | 造测次需的用例 | 设次需考点（低权重 ≤5） |
| 6 行为准则/红线 | 写约束和红线指令 | 造引诱违规的负面用例 | 设安全/合规扣分点 |
| 7 输出规格 | 写格式模板 | — | 设格式考点 |
| 8 多轮行为 | 写多轮行为指令 | 造多轮会话用例 | 设轮间一致性/个性化考点 |
| 9 语气风格 | 写语气段 | — | 设语气符合度考点 |
| 10 备选模型 | — | — | 适配模型容忍度；Eval Runner 据此调 API |
| 11 成功标准 | 理解理想态 | — | 设整体质量基线（通过阈值） |
| 12 典型示例 | few-shot 素材来源 | 必须纳入评测集 | 用于校准评分尺度 |

## 四、交互形态如何驱动结构

`Section 0` 的"交互形态"字段是开关：

| 交互形态 | Section 8 多轮行为 | Eval Generator 影响 | Eval Designer 影响 |
|---------|-------------------|---------------------|-------------------|
| 单轮 | 留空（写"本 Prompt 为单轮，无多轮要求"） | 只造单轮用例 | 不设多轮考点 |
| 多轮 | 必填全部子项 | 必须造多轮会话用例（带历史上下文） | 必须设轮间一致性、个性化、上下文记忆考点 |
| 两者兼容 | 必填，并说明单轮模式下的降级行为 | 单轮和多轮用例都造 | 单轮多轮考点都设 |

下游 Agent 读 PRD 第一件事就是看 Section 0 的交互形态，据此决定自己的工作范围。

---

## 五、完整性校验清单（Phase 1 → 2 门禁）

Requirements Clarifier 在请求用户确认 PRD 前，必须自检以下项。任一不通过不得进入 Phase 2：

- [ ] Section 0 元信息完整，交互形态已明确声明
- [ ] Section 2 角色定义具体（不是"一个有用的助手"这种空泛描述）
- [ ] Section 4 能力边界的"能做/不能做/超界处理"三项都有内容
- [ ] Section 5 主需清单非空，且每条主需是具体可验证的（不是"回答得好"）
- [ ] Section 5 次需与主需没有混淆（次需不应包含核心动作）
- [ ] Section 6 至少有一条安全红线（即便是"无特殊红线"也要显式声明）
- [ ] Section 7 输出长度是具体区间，没有"简洁""适中"等模糊词
- [ ] 若交互形态含多轮，Section 8 全部子项已填
- [ ] Section 10 备选模型已指定主力模型
- [ ] Section 11 成功标准说清了"最关心的质量维度"和"绝对不能出现的错误"
- [ ] Section 12 至少有 1 个理想样例 + 1 个反例

---

## 六、PRD 不该写什么

- **不写实现细节**：PRD 描述"要什么"，不描述 Prompt 具体怎么措辞（那是 Prompt Writer 的事）
- **不写评测方法**：PRD 描述"什么算好"，不描述具体怎么打分（那是 Eval Designer 的事）
- **不留模糊词**：所有"可能""也许""差不多"在确认前必须变成确定的选择
- **不写通识要求**：语病、错别字、排版这类通用质量要求不进 PRD（评测体系统一兜底）
