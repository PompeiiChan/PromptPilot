# Rubric 评分点撰写规范

本文件定义 Eval Designer Agent 设计 Rubric 考点时必须遵守的完整规则。

---

## 一、Rubric 的基本结构

每套 Rubric 由以下部分构成：

```
一二级标题（思路骨架，过程性工具）
  └── 评分点（考点列表，核心交付物）
        ├── 得分点（正向加分）
        └── 扣分点（负向减分）
```

### 一二级标题

过程性工具，不是交付字段。作用是：

- 帮助 Eval Designer 从"理想输出"出发构建评分骨架，避免整体思路混乱
- 帮助 Orchestrator 和人类 reviewer 对齐评测思路

标题不出现在最终评分表里，只用于组织考点。

---

## 二、评分点（Scoring Points）撰写规则

### 2.1 六条 P0 规则（必须全部满足）

**P0-1：全面覆盖核心要点，总数不超过 20 条**

评分点必须覆盖原始 Prompt 和 PRD 定义的核心要求。但：

- 完整 ≠ 穷尽：只写影响"主需满足"的核心点，可答可不答的次要点不写
- 维度要全面：不只考内容，也考质量、常见错误
  - **内容考察点**：要求"给到" + "正确"
  - **质量考察点**：要求"价值高"、"逻辑链完整"
  - **常见错误点**：限制 AI 易犯的错误（扣分点）
- 不纳入通识类要求（语病、排版等）

---

**P0-2：原子性——每条只描述一组动作 + 事实**

每条评分点只包含**一个逻辑单元**，不把多个并列要求或多层逻辑写进同一条。

```
❌ 错误：从优缺点两个方面分析 XX 项目，并指出其带来的增益/损害
✅ 正确（拆为两条）：
   - 指出 XX 项目的优点为……（正面事实）
   - 指出 XX 项目的缺点为……（负面事实）
```

原子化 ≠ 碎片化：不重要的内容适当合并，不为"细分"而细分。

```
❌ 碎片化：
   - 正确选项包括 A
   - 正确选项包括 B
   - 正确选项包括 C
✅ 合理合并：
   - 正确选项为 A、B、C
```

---

**P0-3：具体性——动作 + 具体事实，不允许宽泛描述**

评分点必须按照「动作 + 具体事实」的方式撰写：

- 不能是宽泛描述：❌ "明确指出关键技术的工艺"
- 必须包含具体事实：✅ "明确指出 SK 海力士在 16-Hi HBM3E 产品上采用的封装键合技术为 Advanced MR-MUF 工艺"

禁止出现：
- 不需要回溯上文才能明白指什么的**指代词**（"该项目""上述方案"等）
- 可以适用于任何题目的**通用描述**（"明确指出关键技术的工艺"）
- 语言表达、修辞手法、表格形式等**主观风格类**要求

---

**P0-4：事实真实——包含判断对错所需的事实，且事实本身正确**

每条评分点里的事实性信息必须是真实、可验证的。

- 不能要求评测者"再去搜索额外信息"才能判断
- 不能要求额外专业知识补充才能判断
- 如果事实存在争议，在「补充说明」里注明

---

**P0-5：顺序与逻辑链——评分点顺序与理想输出的逻辑顺序一致**

评分点排列要：
- 与理想回复的先后逻辑保持一致
- 体现显式推导链条：把事实点的前置论据也作为评分点写出来
- 呈现「观点 + 证据支撑」的完整证据链

```
✅ 正确（有推导链条）：
   R-01: 结合 XX 平台用户评论，指出 A 酒店具有 XX 亮点，是适合年轻人的推荐
   R-02: 结合 XX 平台用户评论，指出 B 酒店位于核心商圈但性价比高，是适合年轻人的推荐

❌ 错误（无推导链条）：
   R-01: 根据预算和年轻人喜好，推荐 A 酒店、B 酒店
```

---

**P0-6：独立且逻辑自洽——各评分点之间无交叉、无重复、无矛盾**

- 两条评分点不得有 50% 以上的语义重叠
- 评分点之间不能前后矛盾或逻辑冲突
- 每条必须**可以被独立阅读并完成判断**，不交叉引用其他条目

```
❌ 错误（两条评分点说同一件事）：
   R-05: 明确指出 SK 海力士采用 Advanced MR-MUF 工艺
   R-06: 明确指出 16-Hi HBM3E 产品采用的封装技术为 Advanced MR-MUF 工艺

❌ 错误（指代词，需要回溯上文）：
   总结并分析上述影响海南未来房价的关键政策，需明确指出其对该市的具体影响
✅ 正确（自包含）：
   分析影响海南未来房价的关键政策，明确指出海南自贸港政策对陵水度假型房产的影响
```

---

### 2.2 标准明确——零外查原则

评分标准必须直接包含判断所需的一切信息，不依赖外部查询：

| 答案类型 | 写法 |
|---------|------|
| 唯一客观答案 | 直接写出该答案 |
| 答案有限（多个正确选项） | 列举所有正确回答 + 约束"必须在该范围内" |
| 答案无限/无法穷举 | 写出：要求 + 约束条件（正确回答的特征） + 示例（可选） |

```
❌ 错误：结合具体房价数据对当前市场态势做出清晰准确的判断
✅ 正确：结合一项具体房价数据说明当前市场态势，
         如过去12个月区域新房价格稳定在2.7万元/㎡以上
         如二手房2025年7月3.37万/㎡跌至2025年12月3.18万/㎡，整体呈下行趋势
```

---

### 2.3 动态事实点写法

回复包含高时效性内容（时间、价格、数据）时，评分点中必须写出**该时间点的具体数值**：

```
❌ 错误：比较特斯拉超充和高速服务区价格，指出特斯拉超充更经济实惠
✅ 正确：基于2026年1月1日特斯拉超充价格1.8元/度、高速服务区3元/度，
         指出特斯拉超充更经济实惠
```

---

### 2.4 表述清晰——禁止模糊词和主观评价词

| 禁止使用 | 替换为 |
|---------|--------|
| "大约/左右/接近/很多" | 具体数字或明确范围 |
| "字数大概300字左右" | "字数在280-320字之间" |
| "给出恰当举例" | "给出关于xxx，能印证xxx观点的举例" |
| "逻辑清晰/表达流畅" | 具体说明什么样的逻辑结构算清晰 |

---

### 2.5 表述精炼——每条只含直接影响达成度的信息

评分点里不允许出现不影响判断的解释性内容：

```
❌ 错误：xx问题的答案是A，是因为Bxxxxxx，Cxxxxx
（B、C的解释不影响"答案是否正确"的判断，不应出现在评分点里）

✅ 正确：明确指出xx问题的答案是A
（需要解释的内容放到「补充说明」字段）
```

---

### 2.6 补充说明字段

当考点达成要求相对复杂时，在评分点后隔行补充说明。通常包含：

- 对偏主观、概括性、难判断、未量化词汇的进一步细化
- 复杂的复合判断逻辑（且、或、且+或的组合）
- 边界说明（明确定义"不符合要求"的情况）
- 示例（对开放性信息点，给出模型回答的形式帮助判断）
- 影响该条 rubric 的相关客观事实

## 三、赋分规则

### 3.1 得分点赋分（正向）

| 分段 | 含义 | 占比要求 |
|------|------|---------|
| **9–10 分（必答点）** | 缺失则导致结论不完整或不可信的核心事实 | **必须 ≥ 40%** |
| **6–8 分（重要点）** | 显著提升深度、提供关键洞察的事实 | 约 20–50% |
| **5 分及以下（附加点）** | 背景补充、非核心细节丰富 | **必须 ≤ 15%** |

同一分段内，根据要点重要度给出差异性赋分（不能所有必答点都是10分）。

### 3.2 扣分点赋分（负向）

| 分段 | 含义 |
|------|------|
| **-9 到 -10 分** | 严重缺陷 |
| **-6 分及以下** | 轻微缺陷 |

**扣分点占总体比例必须 ≤ 20%**

扣分点表述方式：「出现了错误A。」「出现问题或异常情况B。」

### 3.3 扣分点类型参考

**① 模型特有错误**：专门捕捉 AI 常犯的具体错误（错误判断、归因、逻辑、幻觉事实等）

```
示例题目：为初创公司决策应该选择 SK 海力士还是三星作为 HBM3E 供应商
扣分点：指出 SK 海力士采用 X9 芯片工艺原理
（这是模型幻觉：SK 海力士根本没有采用 X9 芯片工艺原理）
```

**② 行业/新手通病**：该领域常见错误或初级从业者容易犯的错误

```
示例题目：我最近总是肚子疼，但常规检查都正常，该怎么办？
扣分点：模型判断用户仅是消化不良
（专业：需要考虑消化道以外病因及隐匿性器质性疾病）
```

**③ 指令遵循或格式问题**：强调 AI 应该遵循 Prompt 的具体指令

```
示例题目：请为我推荐一部80-90年代的高质量乡村电影
扣分点：模型推荐了非80-90年代的电影
```

## 四、不同 Prompt 类型的框架原则

根据 PRD 中的 Prompt 类型，选择对应的框架原则来构建一二级标题和评分维度：

### How-to 类

核心评测维度：
- 是否对用户进行合理分层（不同类型用户的做法不同）
- 是否先给方向与优先级，再给做法
- 是否给"方法模型"而非只给步骤（可迁移的方法逻辑）
- 是否指出关键变量与分支条件（哪些条件变化导致做法变化）
- 是否提示风险、边界与反模式
- 是否给出结果预期与验证信号

### 分析决策类

核心评测维度：
- 是否结论先行
- 是否给出判断框架（决策逻辑、核心判断维度）
- 是否有条件化分析（不同条件下结论不同）
- 是否有风险与边界意识
- 是否行动导向（给出低成本低风险的第一步）

### 预测类

核心评测维度：
- 是否先给判断再给论证（结论可被引用）
- 是否基于现实状态推演，不从假设直接跳结论
- 是否识别并解释 2–4 个决定性变量
- 是否给出至少两种情景（基准/上行或下行）
- 是否说明预测不确定性来源

### 开放推荐类

核心评测维度：
- 是否开头给出分流式结论
- 是否解释结论背后的维度逻辑（而非只展示细节）
- 是否加入风险提示（哪些结论依赖前置条件）

### 解读类

核心评测维度：
- 是否先澄清命题（避免在错误口径下讲得很完整）
- 是否有清晰的解释主线（背景→机制→表现→边界→判断）
- 是否解释"为什么"而不只是"是什么"
- 是否明确边界、例外与常见误解
- 是否管理可信度（对争议性内容明确区分）

## 五、Eval Designer 的输出要求

基于以上规则，Eval Designer 生成 `docs/eval-rubric.md` 时：

1. **每条评分点必须通过 P0-1 到 P0-6 的自检**，写完后逐条过一遍
2. **赋分比例必须满足**：必答点(9-10分) ≥ 40%，附加点(≤5分) ≤ 15%，扣分点 ≤ 20%
3. **扣分点必须覆盖**：本 Prompt 场景下 AI 最可能犯的 1-3 类典型错误
4. **框架结构**：先确定 Prompt 类型，套用对应框架原则，再逐条撰写评分点
5. **补充说明**：凡涉及开放性判断、复合逻辑、边界情况的评分点，必须有补充说明

### 自检清单（每条评分点写完后过一遍）

- [ ] 这条只描述了一件事（原子性）
- [ ] 包含了具体事实，不是宽泛描述（具体性）
- [ ] 不需要查外部资料就能判断（零外查）
- [ ] 没有指代词，不需要回溯上文（自包含）
- [ ] 和其他条之间没有 50% 以上的语义重叠（独立性）
- [ ] 赋分符合必答点/重要点/附加点的比例要求（赋分合规）
- [ ] 如果有动态事实，写出了具体数值和时间点（时效性）
