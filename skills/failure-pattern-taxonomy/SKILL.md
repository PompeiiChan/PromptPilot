# 失效模式诊断协议

本文件是 Prompt Pilot 迭代闭环的核心协议，连接三个角色：

```
Eval Runner   → 把"哪些用例失败了"归纳成"失效模式 + 根因诊断"
Orchestrator  → 根据根因路由到正确的 Phase（迭代决策）
Prompt Writer → 根据诊断针对性修改，不靠猜
```

没有这个协议，评测结果只是一堆分数；有了它，评测结果变成可执行的修改指令。

---

## 一、三个层次：症状 / 失效模式 / 根因

| 层次 | 定义 | 谁负责 | 例子 |
|------|------|--------|------|
| **症状（Symptom）** | 单条用例上观察到的具体问题 | `overall-scoring` 的问题标签 + `rubric-scoring` 的考点失分 | C-003 内容价值-深度不足 |
| **失效模式（Pattern）** | 同类症状在多条用例上反复出现，构成系统性问题 | 本协议 | "深度不足"在 8/25 条用例出现 |
| **根因（Root Cause）** | 失效模式背后的真正原因，决定该改哪里 | 本协议的诊断决策树 | Prompt 中根本没要求提供数据支撑 |

**关键认知**：症状不等于根因。同一个症状（"深度不足"）可能有完全不同的根因——
- Prompt 没要求深度 → 改 Prompt
- PRD 没定义什么叫"够深" → 改需求
- 模型能力天花板 → 换模型

只看症状去改 Prompt，很容易改错地方。本协议的价值就是**从症状追到根因**。

---

## 二、五类根因（决定路由）

所有失效模式的根因归入且只归入以下五类。根因类别直接决定 Orchestrator 把迭代球踢回哪个 Phase。

| 根因类 | 含义 | 路由 |
|--------|------|------|
| **F-A 指令缺失** | Prompt 里**根本没提**这件事，模型按默认行为走 | → Phase 2（补充指令） |
| **F-B 指令失效** | Prompt **说了但没生效**：表述有歧义、与其他指令冲突、埋得太深被淹没、约束不够强硬 | → Phase 2（改写指令） |
| **F-C 需求缺陷** | PRD 本身就没把这件事定义清楚，导致 Prompt 无从写对 | → Phase 1（澄清需求） |
| **F-D 评测缺陷** | 失败是评测本身的错：用例期望写错了、用例超出 PRD 范围、rubric 标准不合理 | → Phase 3/4（修正评测） |
| **F-E 模型边界** | Prompt 清晰、PRD 清晰、评测合理，模型就是稳定做不到 | → 换模型 / 调参 / 接受局限 |

**为什么按这个顺序排**：从最廉价高频的修复（改 Prompt）到最昂贵的（换模型、返工需求）。诊断时优先排查 F-A/F-B，因为绝大多数失效是 Prompt 问题。

## 三、诊断决策树（Eval Runner 必须执行）

对每一个识别出的失效模式，按顺序回答以下问题，确定根因类：

```
Q1. Prompt 中是否明确包含了对这件事的要求或约束？
    ├─ 否 → 【F-A 指令缺失】→ Phase 2
    └─ 是 → Q2

Q2. 这条要求在 Prompt 里表述是否清晰、无歧义、无冲突、位置足够显著？
    ├─ 否（歧义 / 与别处冲突 / 埋太深 / 语气太弱）→ 【F-B 指令失效】→ Phase 2
    └─ 是 → Q3

Q3. 这个"失败"是不是评测本身的问题？
    （用例期望写错、用例要求超出 PRD 范围、rubric 考点不合理）
    ├─ 是 → 【F-D 评测缺陷】→ Phase 3/4
    └─ 否 → Q4

Q4. PRD 对这件事是否有清晰、可执行的定义？
    （Prompt 写不对，是不是因为 PRD 本身就模糊）
    ├─ 否 → 【F-C 需求缺陷】→ Phase 1
    └─ 是 → Q5

Q5. 在 Prompt 清晰、PRD 清晰、评测合理的前提下，模型是否仍稳定失败？
    └─ 是 → 【F-E 模型边界】→ 换模型 / 调参 / 接受
```

**快捷提示**：如果你审一条失败用例时心里冒出"这明明答对了啊"，优先跳到 Q3 怀疑评测缺陷，不要急着归咎 Prompt。

## 四、常见失效模式库（参考）

以下是高频失效模式的速查表。它是**参考案例**不是封闭清单——遇到新模式按决策树诊断后补充即可。

### 4.1 任务理解类

| 模式 | 识别信号 | 最可能根因 | 典型修复方向 |
|------|---------|-----------|------------|
| 答非所问 | 输出在回答字面问题，但抓错了用户核心诉求 | F-B 或 F-C | Prompt 中明确"先判断用户真实意图再作答"；或回 PRD 澄清核心诉求 |
| 任务跑偏 | 多步任务只做了一部分，或做成了别的任务 | F-A / F-B | 用编号步骤拆解任务，明确每步必做 |

### 4.2 指令遵循类

| 模式 | 识别信号 | 最可能根因 | 典型修复方向 |
|------|---------|-----------|------------|
| 约束被忽略 | PRD 行为约束在多条用例被违反 | F-A（没写进 Prompt）/ F-B（写了但太弱） | 把约束提到 Prompt 显著位置，用强硬措辞 |
| 输出格式不稳定 | 格式时对时错，边界输入时崩溃 | F-B | 给出格式模板而非文字描述；明确各种情况的格式 |
| 长度失控 | 输出长度系统性超标/过短 | F-A / F-B | 明确字数区间（不用"简洁"等模糊词） |

### 4.3 内容质量类

| 模式 | 识别信号 | 最可能根因 | 典型修复方向 |
|------|---------|-----------|------------|
| 深度不足 | "内容价值-深度"标签高频，多为套话 | F-A（没要求深度）/ F-C（没定义深度标准） | Prompt 要求每个论点配数据/案例；或回 PRD 定义"够深"的标准 |
| 覆盖不全 | "丰富度"标签高频，维度单一 | F-A | Prompt 列出必须覆盖的维度 |
| 特异性差 | 换个主体也成立的通用回答 | F-A / F-B | Prompt 要求紧扣输入的具体特征展开 |
| 幻觉/编造 | "真实性"标签高频 | F-B（没强约束）/ F-E（模型固有） | Prompt 要求"不确定就说不知道"；或接入检索/换模型 |

### 4.4 一致性/稳定性类

| 模式 | 识别信号 | 最可能根因 | 典型修复方向 |
|------|---------|-----------|------------|
| 边界输入崩溃 | happy path 正常，边界用例集体失败 | F-A | Prompt 显式定义边界情况的处理（空输入、超长、格式异常） |
| 同义输入漂移 | 相似输入产出质量/风格差异大 | F-B | 收紧指令、减少模型自由发挥空间；few-shot 锚定 |

### 4.5 评测自身类（指向 F-D）

| 模式 | 识别信号 | 路由 |
|------|---------|------|
| 期望写错 | 多条"失败"用例的输出其实合理，是期望值不对 | Phase 3 修用例 |
| 用例超纲 | 用例要求的能力 PRD 根本没定义 | Phase 3 删用例，或 Phase 1 补需求 |
| rubric 过严/过松 | 考点判断标准不合理导致误判 | Phase 4 调 rubric |

## 五、标准输出 Schema（Eval Runner 写入）

Eval Runner 在 `eval/results/<run_id>.json` 的 `agent_instructions.failure_patterns` 字段，必须按以下结构输出。这是 Prompt Writer 和 Orchestrator 唯一信任的输入：

```json
{
  "pattern_id": "FP-01",
  "title": "分析类回答深度不足",
  "symptom": "内容价值-深度",
  "frequency": "8/25",
  "affected_cases": ["C-003", "C-007", "C-011", "C-015"],
  "severity": "high",
  "root_cause_class": "F-A",
  "diagnosis": "Prompt 输出要求中没有任何关于'论点需有数据或案例支撑'的指令，模型默认给出概括性回答。已确认 PRD Section 9 要求'分析要有深度'，但 Prompt 未将其落地为具体指令。",
  "fix_direction": "在 prompt.md 的输出要求部分增加：每个分析论点至少引用一个具体数据、案例或权威来源，禁止只给结论不给依据。",
  "route": "phase-2",
  "example": {
    "case_id": "C-003",
    "input_excerpt": "分析新能源汽车在三四线城市的渗透前景",
    "output_excerpt": "前景广阔，受政策支持，消费者接受度逐步提升……",
    "why_failed": "通篇结论性套话，无任何渗透率数据、补贴政策细节或区域案例"
  }
}
```

**字段规则**：

| 字段 | 要求 |
|------|------|
| `symptom` | 必须取自 `overall-scoring` 的问题标签或 `rubric-scoring` 的考点 ID，不自创 |
| `frequency` | 实际形式 `命中数/总用例数`，是判断 severity 的依据 |
| `severity` | `high`=涉及主需/安全/导致 FAIL；`medium`=影响质量但仍 PARTIAL；`low`=锦上添花 |
| `root_cause_class` | 必须是 F-A~F-E 之一，且必须是走完决策树的结论 |
| `diagnosis` | 必须说清"为什么是这个根因"，引用 Prompt/PRD 的具体位置，不能只复述症状 |
| `fix_direction` | 必须是**可直接执行**的修改方向，落到 Prompt 的具体段落或 PRD 的具体字段 |
| `route` | `phase-1` / `phase-2` / `phase-3` / `phase-4` / `model-change` / `accept` |
| `example` | 至少给一个最典型的失败用例，含输入/输出摘录和失败原因 |

## 六、Orchestrator 如何消费（迭代路由聚合）

Orchestrator 拿到全部 `failure_patterns` 后，按 `route` 聚合，决定本轮迭代主线：

```
1. 若存在 route=phase-1 的 high severity 模式
   → 优先回 Phase 1（需求有根本缺陷，改 Prompt 是徒劳）

2. 否则若存在 route=phase-3/phase-4 的模式
   → 先修评测（评测不可信时，其他诊断都不可信）

3. 否则收集所有 route=phase-2 的模式
   → 打包传给 Prompt Writer，一次迭代解决多个 Prompt 问题

4. 仅剩 route=model-change/accept 的模式
   → 向用户报告：Prompt 已优化到位，剩余问题是模型边界，请决策换模型或接受
```

**原则**：上游根因优先于下游。需求缺陷 > 评测缺陷 > Prompt 缺陷 > 模型边界。不要在需求模糊时反复打磨 Prompt。

---

## 七、Prompt Writer 如何消费

Prompt Writer 收到 `route=phase-2` 的失效模式包后：

1. 按 `severity` 从高到低处理
2. 每处修改对应一个 `pattern_id`，在迭代说明里标注："针对 FP-01（深度不足，8/25）：在输出要求增加了数据支撑指令"
3. 区分 F-A 和 F-B 的改法：
   - **F-A（缺失）**：新增指令
   - **F-B（失效）**：改写现有指令——消歧、解冲突、提位置、加强语气，而不是简单重复
4. 若发现某个 `route=phase-2` 的模式其实改不动（Prompt 已经写得很清楚了），反馈 Orchestrator 重新诊断，可能是误判的 F-E

---

## 八、这个协议解决的核心问题

没有本协议时：

```
Eval Runner: "内容深度不太够" → Prompt Writer: "那我加一句'请深入分析'" → 没用
```

有本协议时：

```
Eval Runner: FP-01 / 深度不足 / 8 个用例 / 根因 F-A / Prompt 完全没要求数据支撑 / 建议在输出要求加"每个论点配数据案例"
Prompt Writer: 精确定位，加对地方，下一轮验证
```

诊断必须追到根因，修改必须落到具体位置——这是本协议不可妥协的两条底线。
