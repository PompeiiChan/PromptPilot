Prompt Pilot

人机协同驾驭 Prompt 工程

把模糊需求变成生产级 Prompt，用系统化评测驱动持续迭代。
让 Prompt 工程可控、可度量、可交付。

项目背景

为什么 Prompt 工程需要更好的方法

🔴 现状问题

传统的 Prompt 工程是临时的、不可预测的。开发者凭直觉编写 Prompt，手动测试几个例子后就上线—— 只在生产环境才发现边缘问题。没有系统化的质量评估方法，迭代更是靠运气。

🟢 解决方案

Prompt Pilot 将 Prompt 视为软件产品。从需求澄清到自动化评测，每一步都结构化、可度量。 人类做战略决策，AI 负责执行。质量由数据驱动，而非猜测。

核心特性

Prompt Pilot 的与众不同之处

👤

人在回路（HITL）

每个关键决策点都需要人类判断。AI 执行分析，人类做决策。 在保证质量的同时维持效率——兼顾两者的优势。

📊

数据驱动迭代

评测报告包含 Good/Bad case 和根因分析，指导你的下一步行动。 停止猜测——基于证据迭代。

🔄

完整工作流

七阶段从想法到生产：需求澄清 → 编写 → 评测 → 迭代 → 发布。 无遗漏，无死角。

⏮️

Git 版本回滚

每个阶段和每次评测都会自动打标签。可随时回滚到任意历史状态。 大胆实验——总能回到过去。

⚡

并行执行

测试用例生成和评分框架设计并行运行。当步骤相互独立时，无需等待。

🎯

根因分析

失效模式被聚合和分类。精确知道该修复需求、优化 Prompt 还是调整评测—— 上游问题优先处理。

设计理念

Prompt Pilot 背后的设计原则

上游问题优先

不要在需求模糊时反复打磨 Prompt。修复上游需求比迭代下游更有效。 根因分析优先处理 Phase 1 问题而非 Phase 2——先解决正确的问题。

人工判断高于自动化

AI 擅长执行，但战略决策需要人类判断。HITL 协议确保人类审批 PRD、 审查评测报告、决策迭代路线。没有监督的自动化是风险。

度量驱动改进

无法度量就无法改进。基于 Rubric 和测试用例的评测提供客观指标。 质量评分和失效模式告诉你应该聚焦哪里。直觉有帮助，数据更可靠。

通过版本控制保障安全

基于 Git 的标签意味着每个状态都可恢复。可回滚到任意阶段或评测点。 大胆探索——错误可逆转。这个安全网鼓励实验，防止恐惧驱动的保守。
